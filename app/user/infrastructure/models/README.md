# Models 模块说明

`app/user/infrastructure/models/` 放的是仍然由 MySQL 持久化的 SQLAlchemy 模型。这里的重点不是表达完整聊天过程，而是保存少量需要落库的核心实体。

## 结构分工

- `user.py`
  - 负责用户基础信息。
  - 当前主要作用是作为会话归属的外键目标。
- `conversation.py`
  - 负责会话元信息，例如标题、创建时间、状态和对话类型。
  - 不保存逐条消息正文，消息内容由记忆层和缓存层负责。
- `__init__.py`
  - 集中导出当前仍在使用的模型，方便建表脚本和服务层统一导入。
  - `Base` 由 `core/database.py` 统一提供，模型文件本身只关心字段和关系定义。

## 当前边界

- `models/` 只定义表结构和 ORM 关系，不承载查询流程、业务规则或 HTTP 协议细节。
- 如果某段逻辑需要决定“该查什么 / 该怎么更新 / 是否需要事务”，它应该放在应用层或基础设施层，而不是模型文件里。
- 当前项目的持久化边界比较克制：会话只存元信息，不把整段聊天历史重新塞回 MySQL。
- 当前模型已统一成 SQLAlchemy 2 的类型化写法，字段和关系类型应直接在模型层表达清楚，不要再混回无类型的旧式 `Column(...)` 风格。

## 后续维护建议

- 新增字段前，先确认它是否真的属于“需要长期持久化的结构化数据”，避免把缓存型或中间态数据误塞进表结构。
- 如果某个模型文件开始出现复杂 helper、查询拼接或跨表业务判断，说明职责已经外溢，应下沉到服务层。
